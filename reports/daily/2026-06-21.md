## Summary

Today focused on turning the COAD Cancer Predictor project into a public-facing, read-only research website. The website requirements were written and expanded, the old database-console style website was redesigned into a polished COAD project website, the backend was moved to Node.js + TypeScript + Express, and the app was deployed through the Jupyter container on port `3000`.

Additional scientific reporting work also appears in today's project outputs: an AlphaFold hotspot structure report was generated for selected COAD mutation hotspots. AlphaFold is a tool that predicts protein structure, meaning the likely 3D shape of a protein. A hotspot means a mutation position that appears repeatedly in multiple samples.

## What Was Learned

- A public project website should use curated summaries instead of raw database access.
- A high school research website needs clear explanations for both biology terms and computer science terms.
- AlphaFold is useful for showing protein structure context, but confidence scores must be shown so users know when a predicted region is uncertain.
- A polished scientific website needs multiple passes of typography, spacing, responsive layout, and reference-image matching.
- Running the Node app inside the Jupyter container is the right project pattern because the container already exposes the Node service port.

---

## Issues / Limitations

- Some generated visual concepts contained placeholder numbers, so implementation must use project-curated data instead of copying concept-image values.
- The website is still using curated JSON summaries first; the future `coad_project_web` PostgreSQL schema is planned but not fully implemented yet.
- The site is read-only and does not store contact messages; it opens an email flow instead.
- AlphaFold structures are predicted structures, not experimentally measured structures.
- Existing `.DS_Store` files and other unrelated local noise remain in the working tree.
